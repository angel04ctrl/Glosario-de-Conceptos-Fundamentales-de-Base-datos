Conceptos fundamentales en Bases de Datos

Transacciones y Concurrencia

Transacción

Una transacción en sistemas de bases de datos es una unidad lógica de trabajo compuesta por una o varias operaciones (lecturas y escrituras) que deben ejecutarse de forma íntegra y coherente; el objetivo es que la base de datos pase de un estado consistente a otro estado consistente aunque ocurran fallos, errores o accesos concurrentes. Las transacciones facilitan la gestión de cambios porque se comportan como operaciones “todo o nada”: o bien todas las acciones se confirman (commit) y persisten en la base de datos, o bien se deshacen (rollback) dejando la base de datos exactamente como estaba antes de iniciar la transacción. Este comportamiento es esencial en aplicaciones críticas (bancos, comercio electrónico, sistemas de inventario) donde deben garantizarse la exactitud y la integridad de los datos ante fallas o accesos simultáneos.

Referencia (APA 7): Gray, J., & Reuter, A. (1992). Transaction processing: Concepts and techniques. Morgan Kaufmann.

Propiedades ACID

Las propiedades ACID definen las garantías básicas que debe ofrecer una transacción correcta: Atomicidad asegura que todas las operaciones dentro de la transacción se completen como un todo o se reviertan por completo; Consistencia exige que una transacción lleve a la base de datos de un estado válido a otro estado válido respetando todas las restricciones y reglas del esquema; Aislamiento garantiza que las transacciones concurrentes se ejecuten sin interferir de forma que el resultado sea equivalente al de una ejecución serial (una tras otra); y Durabilidad garantiza que, una vez que una transacción se confirma, sus efectos permanecen persistentes aun si ocurre un fallo del sistema. Estas propiedades, formalizadas y difundidas por investigadores como Jim Gray, constituyen el marco teórico que guía el diseño de mecanismos de control de concurrencia, recuperación y registro en los sistemas gestores de bases de datos.

La atomicidad garantiza que una transacción se ejecute como una única unidad indivisible: o todas sus operaciones se aplican (commit) o ninguna se aplica (rollback). Si ocurre un fallo en medio de la transacción, el sistema debe deshacer todos los cambios parciales y dejar la base de datos tal como estaba antes de empezar la transacción. En la práctica esto se consigue con mecanismos de registro (write-ahead log): antes de cambiar la base de datos, el SGBD escribe en un log las operaciones que se van a realizar; si hay fallo, el sistema usa el log para revertir o completar las operaciones pendientes.

La consistencia exige que una transacción lleve la base de datos de un estado válido a otro estado válido respetando todas las reglas y restricciones del esquema: claves primarias/foráneas, restricciones de unicidad, reglas de negocio, triggers, etc. No garantiza automáticamente que la aplicación haga la lógica correcta, pero sí que las restricciones declaradas (por ejemplo, que no existan pedidos sin cliente) se mantengan. Si una transacción viola una restricción, no debe poder confirmarse; el SGBD aborta la transacción y restaura el estado consistente.

El aislamiento asegura que las transacciones concurrentes no interfieran entre sí de forma que el resultado final sea indistinguible de alguna ejecución serial (una por una). En la práctica existen niveles de aislamiento (ANSI SQL): Read Uncommitted, Read Committed, Repeatable Read y Serializable. Cada nivel ofrece un equilibrio distinto entre consistencia y rendimiento: por ejemplo, Read Committed evita lecturas sucias pero permite lecturas no repetibles; Serializable proporciona el nivel más fuerte (equivalente a ejecución serial) pero es más costoso. Los SGBD implementan el aislamiento mediante bloqueos (two-phase locking) o MVCC (control de concurrencia multiversión), que permiten leer versiones consistentes sin bloquear escrituras o viceversa dependiendo del diseño.

La durabilidad garantiza que, una vez que una transacción ha sido confirmada (commit), sus efectos permanecerán permanentemente almacenados incluso si el sistema sufre una caída o fallo de hardware. Para lograrlo los SGBD usan técnicas como el write-ahead logging y el flush/fsync al disco: primero se asegura que el log de la transacción esté físicamente almacenado en disco antes de reportar éxito al cliente; después, el SGBD aplica los cambios a las estructuras de datos en disco y/o en memoria. En caso de reinicio, el proceso de recuperación usa el log para rehacer (redo) las transacciones confirmadas y deshacer (undo) las incompletas.

Bloqueos y concurrencia

El control de bloqueos y concurrencia es el conjunto de técnicas que permiten a múltiples usuarios o procesos acceder simultáneamente a la base de datos sin producir inconsistencias. Un enfoque clásico es el uso de bloqueos (locks) en niveles de granularidad diversos (fila, página, tabla) para evitar lecturas o escrituras conflictivas; por ejemplo, un bloqueo exclusivo impide que otros procesos lean o escriban el recurso mientras está en uso, y un bloqueo compartido permite lecturas concurrentes pero bloquea escrituras. Además de los bloqueos, existen mecanismos como el control de concurrencia multiversión (MVCC), esquemas de timestamp y protocolos de dos fases (2PL) que buscan maximizar rendimiento y minimizar esperas o bloqueos mutuos (deadlocks). La correcta elección y afinamiento de estas técnicas es crucial para obtener un buen rendimiento en cargas altas y para garantizar la integridad de los datos frente a accesos concurrentes.
